正文（纯文本，可直接粘贴公众号）

HAMi 的 GPU 虚拟化不依赖驱动改动，通过纯用户态的 CUDA API 劫持，实现多容器共享一块物理 GPU，并严格隔离显存和算力配额。 一、资源请求 资源 key 按厂商独立配置。以 NVIDIA 为例，用户在 Pod 里写三个字段：nvidia.com/gpu 指定卡数，nvidia.com/gpumem 指定显存上限（MB），nvidia.com/gpucores 指定算力百分比（0-100）。Cambricon 对应 cambricon.com/vmlu，Hygon 对应 hygon.com/dcunum，华为昇腾对应 huawei.com/Ascend910A，均可通过 ConfigMap 覆盖默认值（config/config.go:288）。这些数字最终会变成容器内的环境变量，驱动运行时拦截机制生效。 二、准入与调度 Pod 提交时，MutatingWebhook（webhook.go:81）先拦截，为有 GPU 请求的 Pod 注入调度器名，并调用各设备类型的 MutateAdmission() 标准化资源字段。 进入调度阶段，hami-scheduler 作为 kube-scheduler extender 被调用。Filter 接口（scheduler.go:738）内部对全部候选节点跑 Fit()（device.go:733）六项检查：健康状态、GPU 型号过滤、UUID 过滤、时间片是否耗尽、剩余显存是否够用、剩余算力核心是否够用。筛选和打分完成后，Filter 直接选出最优节点，调用 PatchAnnotations()（scheduler.go:795）把选定设备写入 Pod 的两个注解：hami.io/vgpu-devices-to-allocate（device plugin 后续读取）和 hami.io/vgpu-devices-allocated（状态记录），然后只把这一个节点返回给 kube-scheduler。Bind 接口（scheduler.go:667）随后负责加 node lock 并调用 k8s Pods.Bind() API 完成实际节点绑定，不再改动设备注解。 三、设备分配 kubelet 触发 Device Plugin 的 Allocate()（server.go:593）。它从 Annotation 里读出选定设备，注入三个关键环境变量：CUDA_DEVICE_MEMORY_LIMIT_{i} 设定每张卡的显存上限，CUDA_DEVICE_SM_LIMIT 设定算力百分比，CUDA_DEVICE_MEMORY_SHARED_CACHE 指向共享缓存文件路径。 同时挂载两个文件进容器：libvgpu.so（拦截库本体）和 /etc/ld.so.preload（让 Linux 动态链接器在任意进程启动前优先加载该库）。若容器设置了 CUDA_DISABLE_CONTROL=true 则跳过 preload 注入，容器可完全退出 HAMi 的资源管控（server.go:694-711）。 四、运行时隔离原理 容器启动后，LD_PRELOAD 让 libvgpu.so 最先被加载，它通过 dlsym 重写以 cu 和 nvml 开头的 NVIDIA 库函数入口。 显存隔离：拦截 cuMemAlloc 系列函数，若当前已用显存加申请量超过 CUDA_DEVICE_MEMORY_LIMIT_{i} 设定的配额，直接返回 CUDA_ERROR_OUT_OF_MEMORY；拦截 nvmlDeviceGetMemoryInfo，使 nvidia-smi 只显示配额值而非物理真实值。 算力限流：拦截 cuLaunchKernel，每次提交 kernel 前消耗全局计数器 g_cur_cuda_cores，计数器耗尽时进程自旋等待；后台的 utilization watcher 线程持续测量实际 GPU 利用率，动态补充计数器，将容器算力稳定控制在 CUDA_DEVICE_SM_LIMIT 设定的百分比以内。 五、精妙与局限 这套机制对容器内所有进程透明生效，无需修改环境变量，对 nvidia 驱动零侵入，仅靠用户态库完成了显存和算力的双维度隔离。代价是：算力限制依赖自旋轮询，存在短时超配；依赖 LD_PRELOAD，静态链接的程序理论上可绕过管控。
