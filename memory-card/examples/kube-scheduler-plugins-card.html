正文（纯文本，可直接粘贴公众号）

kube-scheduler 的调度策略都以插件实现，默认启用哪些、什么顺序，答案在 default_plugins.go，基于 v1.37 开发分支核对。 一、默认插件从哪来 默认插件全部经 MultiPoint 注册，基础 18 个，再按 feature gate 追加 DynamicResources 和 NodeDeclaredFeatures，共 20 个。框架初始化时把插件展开到各自实现的扩展点，点内顺序就是注册顺序。 二、入队 Pod 进入队列前要过 PreEnqueue 门禁：SchedulingGates 检查调度门控，DynamicResources 确认 ResourceClaim 就绪，DefaultPreemption 拦截异步抢占中的 Pod。 任一不放行就被门控进 unschedulableEntities 等事件重试，失败重新入队也要再过一遍。入队后由 PrioritySort 按优先级排序，同级按时间。 三、调度周期 调度周期串行执行。PreFilter 11 个插件做预处理和快速预判，Filter 14 个插件逐节点并发过滤，一票否决。 全部节点被滤掉时进 PostFilter：先由 DynamicResources 释放空闲 ResourceClaim，再由 DefaultPreemption 抢占，释放比驱逐代价低。 可行节点进入打分。PreScore 7 个插件准备状态，Score 9 个插件打分加权求和：TaintToleration 权重 3，NodeAffinity、PodTopologySpread、InterPodAffinity、DynamicResources 为 2，其余为 1。 选出最高分节点，assume 在缓存预占资源，Reserve 预留卷与设备，Permit 默认无插件直接放行。 四、绑定周期 绑定要调 API Server，放在独立 goroutine 异步执行，主循环立即处理下一个 Pod，这正是 assume 预占的意义。 绑定周期依次是 WaitOnPermit、PreBind（VolumeBinding 绑定 PV，DynamicResources 写分配结果）、Bind（DefaultBinder 写 pods/binding）、PostBind，失败则 Unreserve 回滚回 backoffQ。 对比旧清单有三处变化：DynamicResources 横跨七个扩展点；DefaultPreemption 新增 PreEnqueue；NodeDeclaredFeatures 加入默认过滤。
